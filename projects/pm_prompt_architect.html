The Architecture

Every perfect prompt requires 5 Dimensions.

Layer 01

Role & Context

Who you are, company stage, and product type. A seed-stage startup PM and a Series-D enterprise PM require fundamentally different documentation structures, engineering guardrails, and tones. We enforce context before constraints.

Layer 03

Style & Aesthetic Core

This is where vague terms like "make it look premium like Apple" are intercepted and translated mathematically. We extract:

  • Inter / SF Pro font stacks
  • 300ms cubic-bezier transitions
  • Extreme whitespace padding variables
  • Monochromatic contrast ratios
Layer 02

Goal Delivery

The structural backbone. What exact output is needed, mapped directly to success criteria.

Layer 04

Tech Constraint

Targeting Cursor needs file-tree logic. v0.dev needs Tailwind utility classes. The prompt adapts.

Layer 05

Audience & Edge Cases

Separating a toy demo from a production-ready feature. Who sees this? Anticipating empty states, responsive bounds, error fallbacks, and user latency expectations.

System Design

The 5-Step Intelligence Pipeline.

Not a template stitcher. This is an autonomous reasoning engine built into the LLM system prompt that analyzes user input before generating a single line of code.

1
Interpret
Translate vague PM desires into technical specs.
2
Detect Gaps
Identify missing UX pieces (empty states, CTA) and inject defaults.
3
Anti-Patterns
Spot UI contradictions (e.g., brutalist + soft shadows).
4
Localize
Format purely for the target tool engine (Cursor vs v0).
5
Self-Check
Validate against the original 5 requirements.
PM Execution

What this proves.

Key PM Decisions Made

I initially tried building a rigid API backend to combine prompt layers. Validation via Grok and Claude stressed that use cases varied too wildly. So, I pivoted heavily towards a Prompt Sensitivity Filter—abandoning simple brainstorming tasks to focus 100% on high-value, high-complexity tasks (prototypes, competitive tear-downs, PRDs).
